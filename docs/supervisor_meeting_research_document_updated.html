1. Executive Summary

The research project develops a vehicle type-based accident risk prediction and alert system for hazardous hill-country roads. Kadugannawa was selected as the main case study because it contains steep gradients, sharp curves, changing weather, limited visibility conditions, and mixed traffic involving cars, vans, buses, lorries, motorcycles, and three-wheelers.

The main research idea is that different vehicle types do not face equal accident risk under the same road and weather conditions. A wet sharp bend at night can be more dangerous for a motorcycle or loaded lorry than for a small car. Therefore, the system combines historical accident patterns, road geometry, weather, time, vehicle type, and live telemetry to predict a Low, Medium, or High risk level and provide a vehicle-specific warning.

Current overall status: The project now contains an end-to-end research prototype: accident/weather/road data collection, preprocessing, feature engineering, Random Forest and XGBoost experiments, baseline model comparison, SMOTE imbalance handling, SHAP explainability, backend prediction APIs, weather and segment services, a web dashboard, Android authentication and mobile prediction integration, real-time driving-session map, OpenFreeMap/MapLibre rendering, risk-zone overlays, trip start/end flow, foreground GPS tracking, and telemetry upload/offline queue handling.

2. Research Background

Mountain-road accident prediction is different from general urban accident prediction because terrain and micro-climate conditions have strong influence. Kadugannawa roads include slopes, bends, rain exposure, fog/visibility variation, and heavy vehicle movement. Traditional warning systems generally provide static road signs or generic alerts. The proposed system moves toward dynamic, context-aware alerts that are calculated from current and historical conditions.

The research contribution is both analytical and practical. Analytically, it investigates which features are most useful for predicting road accident severity in a mountainous environment. Practically, it implements a deployable prototype where mobile users can select a vehicle type, start a trip, view their location on a free map, receive periodic risk predictions, and send GPS/telemetry data to the backend.

3. Research Problem, Questions, and Objectives

Problem Statement

Existing road accident prediction systems often treat all vehicles similarly, focus on urban or highway environments, and lack integration between vehicle type, real-time weather, road geometry, and live driving telemetry. This creates a gap for hill-country roads where terrain, visibility, and vehicle-specific handling characteristics can significantly change accident risk.

Research Questions

  1. What machine learning algorithms have been used for early road accident prediction?
  2. Which mountain-road, weather, and vehicle-specific factors influence accident risk?
  3. How can an accurate machine learning model be developed to identify accident risk in advance for mountainous roads?
  4. How can the model be evaluated and integrated into a real-time warning system?

17. Real-Time Driving Flow

The live mobile flow now matches the research goal: a driver selects a vehicle type before starting a trip, the app tracks GPS, finds nearby road risk zones, requests periodic predictions, and displays warning information while driving.

  1. User signs in and opens the driving map screen.
  2. App loads road risk zones from /api/mobile/risk-zones.
  3. User selects vehicle type: car, van, bus, lorry, or motorcycle.
  4. User taps Start; app requests location permission if needed.
  5. Foreground LocationService starts and waits for GPS fix.
  6. On first GPS fix, app calls /api/mobile/trip/start with vehicle type and start location.
  7. Every 15 seconds, MapViewModel builds a prediction request using vehicle type, location, speed, heading, nearest zone gradient, nearest zone curvature, hour, and day of week.
  8. Backend returns risk level, risk score, probabilities, recommended speed, and alert message.
  9. Map UI updates the risk badge, alert message, recommended speed, route line, current location, score, speed, and distance.
  10. Telemetry worker uploads GPS and movement context to /api/telemetry or stores it in the offline queue for later sync.
  11. User taps End; app calls /api/mobile/trip/end with end location, travelled distance, and duration.
Map implementation decision: The mobile map uses a WebView with Leaflet, MapLibre, and OpenFreeMap style tiles at https://tiles.openfreemap.org/styles/liberty. This avoids a Google Maps key requirement and avoids relying on blocked OpenStreetMap volunteer tiles.

Potential data leakageOutcome-derived fields can inflate performance.Continue excluding injuries, fatalities, generated risk score, and other post-accident outcome fields.
Weather granularityDaily weather may not match exact accident time.Use hourly/current API weather in real deployment and document current granularity.
OSM road geometry qualitySegment slope/curvature may differ from real road conditions.Manually validate high-risk segments and compare with local knowledge when possible.
Mobile deployment uncertaintyGPS, permissions, network, and background restrictions can affect live use.Use foreground service, offline queue, retry logic, cached/default context values, and device testing.
Ethics and privacyPolice data and GPS data contain sensitive information.Anonymise records, avoid storing unnecessary personal data, and explain GPS privacy in the dissertation.
